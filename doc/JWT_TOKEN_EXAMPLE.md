# JWT Token Example

## Token Structure

The JWT generated by this system contains only two claims in the payload:

### Decoded JWT Payload
```json
{
  "id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "expiresAt": 1733270400000
}
```

### Field Descriptions

**id** (String)
- A randomly generated UUID (v4)
- Correlates to a user in the MongoDB database
- Does NOT contain any user information (username, email, etc.)
- Rotates on every token generation
- Used to look up the associated user and validate token status

**expiresAt** (Long)
- Expiration timestamp in epoch milliseconds
- Example: `1733270400000` = December 4, 2024 00:00:00 GMT
- Explicitly checked during validation
- Set based on user's chosen expiration duration (1 day to 3 months)

## Complete JWT Example

A complete JWT token consists of three parts separated by dots:

```
eyJhbGciOiJIUzI1NiJ9.eyJpZCI6ImExYjJjM2Q0LWU1ZjYtNzg5MC1hYmNkLWVmMTIzNDU2Nzg5MCIsImV4cGlyZXNBdCI6MTczMzI3MDQwMDAwMH0.signature_here
```

### Parts Breakdown

1. **Header** (Base64URL encoded)
   ```json
   {
     "alg": "HS256"
   }
   ```
   - Algorithm: HMAC-SHA256

2. **Payload** (Base64URL encoded)
   ```json
   {
     "id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
     "expiresAt": 1733270400000
   }
   ```

3. **Signature**
   - HMAC-SHA256 signature using the server's secret key
   - Ensures token integrity and authenticity

## Why Only Two Fields?

### 1. **Privacy**
- No user information exposed if token is intercepted
- The `id` UUID reveals **absolutely nothing** about the user
- Not the username, not the email, not the real name, not any personal data
- User correlation exists **only** in the server's database
- Even the server admin can't identify a user from just looking at the UUID
- Must query the database to find which user the UUID belongs to

### 2. **Minimal Size**
- Smaller tokens = less bandwidth
- Important for UDP relay where every byte counts
- Typical token size: ~150-180 bytes

### 3. **Security**
- Token ID rotation on each generation
- Database lookup allows immediate revocation
- Expiration is explicit and verifiable

### 4. **Simplicity**
- Easy to validate
- Clear separation of concerns (token vs. database)
- No redundant data (user info is in database)

## Validation Process

When a client sends a JWT token:

1. **Signature Verification**
   - Verify HMAC-SHA256 signature using secret key
   - Ensures token wasn't tampered with

2. **Extract Claims**
   - Parse `id` and `expiresAt` from payload

3. **Check Expiration**
   - Compare `expiresAt` with current time
   - Reject if expired

4. **Database Lookup**
   - Query MongoDB for `TokenInfo` by `id`
   - Verify token is not revoked
   - Get associated user ID

5. **User Validation**
   - Query MongoDB for `User` by user ID
   - Verify user is active
   - Verify this token ID matches user's current token

6. **Authorization**
   - If all checks pass: Allow proxy access
   - If any check fails: Reroute to HTTPS

## Comparison with Standard JWTs

### Standard JWT (e.g., OAuth2)
```json
{
  "sub": "user@example.com",
  "name": "John Doe",
  "iat": 1516239022,
  "exp": 1516242622,
  "aud": "my-app",
  "iss": "auth-server",
  "roles": ["user", "admin"]
}
```

### Illiad Proxy JWT (Minimal)
```json
{
  "id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "expiresAt": 1733270400000
}
```

### Why Different?

| Aspect | Standard JWT | Illiad JWT | Reason |
|--------|-------------|------------|---------|
| User Info | Yes (sub, name, email) | No (only UUID) | Privacy - no PII in token |
| Size | Large (~300-500 bytes) | Small (~150-180 bytes) | Efficiency for UDP |
| Revocation | Difficult (stateless) | Easy (database lookup) | Security requirement |
| Issued At | Yes (iat) | No | Not needed - creation tracked in DB |
| Audience | Yes (aud) | No | Single-purpose token |
| Issuer | Yes (iss) | No | Single issuer (this server) |
| Roles | Often included | No | Not needed for binary auth |

## Token Lifecycle Example

### 1. User Generates Token (30-day expiration)
```bash
curl -X POST "https://example.com/api/auth/token/generate?userId=507f1f77bcf86cd799439011" \
  -H "Content-Type: application/json" \
  -d '{"expirationMinutes": 43200}'
```

Response:
```json
{
  "success": true,
  "message": "Token generated successfully",
  "data": {
    "token": "eyJhbGciOiJIUzI1NiJ9.eyJpZCI6ImExYjJjM2Q0LWU1ZjYtNzg5MC1hYmNkLWVmMTIzNDU2Nzg5MCIsImV4cGlyZXNBdCI6MTczNTg2MjQwMDAwMH0.Xh7j9K2mN4pQ6rS8tU0vW1xY2zA3bC4dE5fF6gG7hH8",
    "expirationMinutes": "43200"
  }
}
```

### 2. Server Creates Database Records

**TokenInfo Collection:**
```json
{
  "_id": ObjectId("..."),
  "userId": "507f1f77bcf86cd799439011",
  "tokenId": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "createdAt": ISODate("2024-12-03T00:00:00Z"),
  "expiresAt": ISODate("2025-01-02T00:00:00Z"),
  "revoked": false
}
```

**User Collection (updated):**
```json
{
  "_id": ObjectId("507f1f77bcf86cd799439011"),
  "username": "user123",
  "currentTokenId": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  ...
}
```

### 3. Client Uses Token

Client includes JWT in Illiad header when connecting to proxy.

### 4. Token Expires (30 days later)

- Token's `expiresAt` is in the past
- Validation fails at step 3 (expiration check)
- Connection rerouted to HTTPS
- User must generate new token

### 5. User Generates New Token

- Old token ID: `a1b2c3d4-e5f6-7890-abcd-ef1234567890`
- New token ID: `f9e8d7c6-b5a4-3210-9876-543210fedcba`
- User's `currentTokenId` updated to new ID
- Old token immediately invalid (even if not expired)

## Security Notes

1. **Never log JWT tokens** - They are credentials
2. **Use HTTPS for API** - Protect token during generation/download
3. **Rotate secret key periodically** - Invalidates all tokens
4. **Monitor failed validations** - Detect attack attempts
5. **Rate limit token generation** - Prevent abuse
6. **Secure MongoDB** - Use authentication and encryption
7. **Backup token database** - For audit trails
